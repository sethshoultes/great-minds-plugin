## What Didn't Work

### 1. Board Review Came Too Late
Fundamental questions — moat, revenue, compounding — should be asked before two rounds of debate, synthesis, creative review, and demo scripting. By the time Jensen asked "What stops someone from copying this?", the answer was locked: nothing.

### 2. "v2 Fantasy" Accumulated Unexamined
Throughout the documents, hard problems were deferred:
- Enterprise configuration: v2
- Caching: v2
- Enforcement: v2
- Internationalization: v2
- Human annotation: v2
- ML classification: v2

Warren Buffett was right: "At some point, v2 becomes a fantasy."

### 3. No Technical Validation Before Full Process
If this was truly a "2-hour feature pretending to be a 2-day feature," why not validate with a 30-minute spike before full debate?

### 4. Creative Reviews Addressed Finished Artifacts
Maya and Jony reviewed code already written. Earlier involvement at outline stage might have prevented issues rather than corrected them post hoc.

## Refined Frameworks

### 1. Front-Load Strategic Questions
Before ideation begins, answer:
- Who pays for this?
- What stops competitors from copying it?
- What compounds over time?

If answers are "no one," "nothing," and "nothing" — proceed with clear eyes this is infrastructure, not product.

### 2. Require Technical Spike Before Full Process
A 2-hour spike should precede multi-day deliberation. Confirm feasibility before investing creative resources.

### 3. Involve Creative Reviewers Earlier
Maya and Jony should see outlines, not finished artifacts. Shape direction, don't just polish endpoints.

### 4. Create v2 Accountability
Any deferral requires:
- An explicit owner
- A target date
- A kill condition if v2 doesn't ship

### 5. Reconcile Conflicting Board Scores
A 4/10 and 8/10 on the same project means evaluators measure different things. Force alignment before verdict.
